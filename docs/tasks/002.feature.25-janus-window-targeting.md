# 002.feature.25 — Janus Window Targeting

## Problem

When puppet's `janusTab()` creates a web panel in Janus, it hits the global HTTP API (`localhost:9223`) and CDP endpoint (`localhost:9222`) with no concept of _which_ Janus window should receive the tab. If multiple Janus windows are open (different projects), the tab may be created in the wrong window — typically whichever is currently focused.

The web panel should be created in the same Janus window that contains the Chat/terminal panel originating the puppet request.

## Current Behavior

```
janusTab({ url: 'http://localhost:3000' })
  → POST http://localhost:9223/api/tabs { url: '...' }
  → Tab created in whichever Janus window the API routes to (likely focused window)
```

All helpers (`janusListTabs`, `janusCloseTab`, `janusNavigateTab`) also hit the global API with no window scoping.

## Desired Behavior

```
janusTab({ url: 'http://localhost:3000' })
  → Automatically detects the originating Janus window
  → POST http://localhost:9223/api/tabs { url: '...', windowId: 'abc123' }
  → Tab created in the correct Janus window
```

## Investigation Needed

### 1. Janus-side: How are windows identified?

- Does Janus assign unique IDs to each window?
- Does the HTTP API already support a `windowId` parameter? If not, it needs one.
- How does `POST /api/tabs` currently decide which window gets the new tab?
- Can `GET /api/tabs` return which window each tab belongs to?

### 2. How can puppet know the originating window?

The most reliable approach is likely an **environment variable** set by Janus when it spawns a terminal/shell:

```
JANUS_WINDOW_ID=abc123
```

When Janus creates a terminal panel in a specific window, it sets this env var in the shell's environment. Then puppet reads it automatically.

**Alternative approaches to evaluate:**

- Query the Janus API for "which window has a terminal with PID X" (complex)
- Pass `windowId` explicitly from the caller (manual, error-prone)
- Use the Janus CDP connection to discover which window the terminal belongs to

### 3. Fallback behavior

When `JANUS_WINDOW_ID` (or equivalent) is not set:

- Fall back to current behavior (global API, no window targeting)
- This preserves backward compatibility for single-window setups

## Implementation Plan (Tentative — Pending Investigation)

### Phase 1: Investigate Janus window model

- Read Janus source to understand window management
- Check if `windowId` concept already exists in the API
- Determine how Janus spawns terminals and what env vars it sets

### Phase 2: Janus-side changes (if needed)

- Add `windowId` support to `POST /api/tabs` endpoint
- Add `windowId` to `GET /api/tabs` response
- Set `JANUS_WINDOW_ID` env var when spawning terminal panels
- Scope `GET /api/tabs`, `DELETE /api/tabs/:id`, `POST /api/tabs/:id/navigate` to window context

### Phase 3: Puppet-side changes

- Read `JANUS_WINDOW_ID` from environment in `janusTab()`
- Add optional `windowId` field to `JanusOptions`
- Pass `windowId` in all Janus API calls
- Auto-detect: `options.windowId ?? process.env.JANUS_WINDOW_ID`
- Update `janusListTabs`, `janusCloseTab`, `janusNavigateTab` to accept/pass `windowId`

### Phase 4: Test

- Multi-window scenario: two Janus windows, verify tab lands in the right one
- Single-window fallback: no `JANUS_WINDOW_ID` set, current behavior preserved
- Explicit override: `janusTab({ windowId: '...' })` targets specific window

## Files Likely Affected

**Puppet:**

- `src/janus.ts` — all helpers need `windowId` support
- `src/types.ts` — add `windowId` to relevant types

**Janus (separate project):**

- HTTP API tab endpoints — accept `windowId` parameter
- Terminal spawning — set `JANUS_WINDOW_ID` env var
- Window management — expose window IDs

## Notes

- This is a cross-project change requiring coordination between puppet and Janus
- The Janus-side changes are a prerequisite — puppet can't target windows if Janus doesn't support it
- The env var approach (`JANUS_WINDOW_ID`) is the most ergonomic since it requires zero code changes from callers (abra, tests, etc.)
